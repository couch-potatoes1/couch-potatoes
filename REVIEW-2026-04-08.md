### Item 1 — Household linking on sign-in *(main item — needs the most scrutiny)*
**Schema (`supabase/schema.sql`):**
- Appended a new `household_members` table at the bottom, marked **DO NOT APPLY YET**. One row per user, columns `user_id` (PK, FK → auth.users), `household_id`, `slot` (1 or 2), `joined_at`, `updated_at`. RLS: user reads/writes only their own row. Index on `household_id` for partner-side lookups.
- Read the comment block above the table — I explain why it's not a hard FK to households, and why "Phase-1 share-link security" still works alongside this.

**Client (`index.html`, near the consent useEffect):**
- New `linkStatus` / `linkConflict` state and a useEffect that fires when `consentStatus === 'ok'` and `authUser` is set.
- Three cases:
  1. **No membership row** → look up the local household, pick the first open slot (slot 1 if `profile1` empty, otherwise slot 2), insert the membership row.
     - **REVIEW edge case:** if both slots are filled and neither belongs to this user, I currently default to slot 1 and `console.warn`. The right answer is probably to spin up a fresh household or ask which slot is theirs — punted because it's a UI question.
  2. **Membership row matches local `cp_householdId`** → no-op.
  3. **Membership row points at a different household** → set `linkStatus = 'conflict'` and surface the new modal.
- New `<HouseholdLinkModal>` JSX block at `z-[70]` (same level as consent — they never show simultaneously because consent gates the linking effect). Two buttons:
  - **"Use my saved household"** → writes `linkConflict.savedHouseholdId` into `localStorage` and `window.location.reload()`. Cheapest way to re-bootstrap everything that hangs off `householdIdRef`. A smoother long-term answer is to plumb a `setHouseholdId()` that re-runs the cloud-sync effects in place.
  - **"Stay on this device"** → just closes the modal. We deliberately do NOT auto-overwrite the membership row to point at the local household — that would silently abandon the partner who might still be using the saved household.
- Sign-out resets the `linkAttemptedRef` guard so the next sign-in re-runs the effect.

**What I tested:** nothing — there's no real way to test this without the migration applied. The whole effect is wrapped in try/catch so the app won't crash if the table doesn't exist; it'll just log "relation does not exist" and `linkStatus` will be `'error'`.

**Before applying the migration:** read the schema comment, decide whether you want to also gate `households` writes on membership eventually (Phase 2). Right now `households` RLS stays public read/write, so this table is purely additive.
